# Cockpit ŌĆö component inventory

A record of the cockpit's components, first extracted from the settled study `cockpit.html`
(per `rules/design-studies.md` "Ship the inventory") and now ŌĆö with the spine assembled (#30) ŌĆö
an OUTPUT of the built screen rather than a contract it must match. The code is the truth; this
file follows it. Any row still at `spec` names a region not yet built; its props are a proposal,
not a frozen law. The study's markup/CSS remain disposable reference. Behavior ŌĆö
Delivery-lifecycle rules R1ŌĆōR16, the session state table S0ŌĆōS11, and panel anatomy ŌĆö lives in
`cockpit-matrix.md`; this file covers only *what* the components are.

Coverage: the study carries **31 distinct `data-component` regions**; each appears below,
but the code ŌĆö not the region count ŌĆö is truth now, so the mapping is no longer one-to-one.
Two study regions were **inlined** rather than extracted ŌĆö `ConsoleChannelTabs` into `Console`
and `ChangesViewToggle` into `DeliveryTabs` (each a single-use block that never justified a file). The build **adds** components the study had no region for:
`LiveTerminal` (a real xterm shell), the `Delivery` panel-body wrapper, and `CiCard` (a shared
card shell); `NodeDrawer` grew from one file into a `NodeDrawer/` directory of per-node bodies. Recurring primitives the study renders without a `data-component` attribute (the
icon helper, dots, chips, buttons, tool/message/check rows, the finding card, the screen
itself) are included and marked **unnamed in study** ŌĆö `/componentize-design` should treat
their names here as proposals to freeze.

Tiering follows `rules/ui-components.md`: atoms ŌåÆ molecules ŌåÆ organisms ŌåÆ screen,
grouped one table per tier.

## Status ŌĆö which artifact is truth

The cockpit is built row by row over many tickets, so `cockpit.html` is authoritative for
some regions and stale for others. The Status column says which, per row:

| Value | Truth | What a change means |
|---|---|---|
| `spec` | The study region | Design change ŌåÆ edit `cockpit.html`. That is what the study is for. |
| `partial ┬Ę <path>` | Both ŌĆö code for what it renders, the study for the rest | Notes name what is still outstanding. |
| `built ┬Ę <path>` | The component + its stories | Change made while implementing ŌåÆ update this row's name/props. Leave the study region alone; it is frozen reference and may be stale. |

Paths are relative to `apps/desktop/src/renderer/src/`: domain components live under
`domains/<region>/components/` and shared primitives under `shared/components/ui/`. The
vocabulary rename (#94) has now **landed in code**, so the earlier `(code currently @ work/ŌĆ” ┬Ę
story/ŌĆ” ┬Ę rail/ŌĆ”)` parentheticals ŌĆö which named the pre-rename on-disk file during the
transition ŌĆö have been removed. Where a Notes cell still carries an old name it is the **study**
anchor (`.srow`, `fn railHtml`, a `data-component`), kept greppable per `rules/design-studies.md`,
not a live path.

**A `built` row's name and props are whatever the code says.** With the spine assembled the
screen is no longer frozen against this file ŌĆö the inventory records the built components, it
does not gate them. Where a build renamed a study proposal, the Notes keep the old name so the
study anchor stays findable.

**Only touch `cockpit.html` for a built region when the drift would mislead a still-`spec`
neighbour** ŌĆö a changed shell, split ratio, or row height that unbuilt regions are laid out
against. Cosmetic divergence inside a built region is expected and costs nothing.

`data-component` regions).

**Text is under everything below.** Wherever a Composed-of column names a word, a label, a
count, a path, a timestamp or prose, that string is rendered by the Text atom ŌĆö the tables
name the copy, not a second way of setting type. The only exceptions are the two primitives
whose `asChild` hands their children to a Slot (Badge, Button); those compose the role class
from Text's map instead of wrapping.
