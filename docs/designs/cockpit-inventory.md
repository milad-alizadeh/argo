# Cockpit — component inventory

A record of the cockpit's components, first extracted from the settled study `cockpit.html`
(per `rules/design-studies.md` "Ship the inventory") and now — with the spine assembled (#30) —
an OUTPUT of the built screen rather than a contract it must match. The code is the truth; this
file follows it. Any row still at `spec` names a region not yet built; its props are a proposal,
not a frozen law. The study's markup/CSS remain disposable reference. Behavior —
Delivery-lifecycle rules R1–R16, the session state table S0–S11, and panel anatomy — lives in
`cockpit-matrix.md`; this file covers only *what* the components are.

Coverage: the study carries **31 distinct `data-component` regions**; each appears below,
but the code — not the region count — is truth now, so the mapping is no longer one-to-one.
The study's `ConsoleChannelTabs` strip is **inlined into the built `Console`** (no separate
component), and the build **adds `LiveTerminal`** (a real xterm shell) which the study had no
region for. Recurring primitives the study renders without a `data-component` attribute (the
icon helper, dots, chips, buttons, tool/message/check rows, the finding card, the screen
itself) are included and marked **unnamed in study** — `/componentize-design` should treat
their names here as proposals to freeze.

Tiering follows `rules/ui-components.md`: atoms → molecules → organisms → screen,
grouped one table per tier.

## Status — which artifact is truth

The cockpit is built row by row over many tickets, so `cockpit.html` is authoritative for
some regions and stale for others. The Status column says which, per row:

| Value | Truth | What a change means |
|---|---|---|
| `spec` | The study region | Design change → edit `cockpit.html`. That is what the study is for. |
| `partial · <path>` | Both — code for what it renders, the study for the rest | Notes name what is still outstanding. |
| `built · <path>` | The component + its stories | Change made while implementing → update this row's name/props. Leave the study region alone; it is frozen reference and may be stale. |

Paths are relative to `apps/desktop/src/renderer/src/components/`. Where this file names a
target module the vocabulary rename (#94) has not yet reached in code, the Notes carry the
current on-disk name as `(code currently …)` so the study anchor and the live file both stay
findable.

**A `built` row's name and props are whatever the code says.** With the spine assembled the
screen is no longer frozen against this file — the inventory records the built components, it
does not gate them. Where a build renamed a study proposal, the Notes keep the old name so the
study anchor stays findable.

**Only touch `cockpit.html` for a built region when the drift would mislead a still-`spec`
neighbour** — a changed shell, split ratio, or row height that unbuilt regions are laid out
against. Cosmetic divergence inside a built region is expected and costs nothing.

## Atoms

All atoms are unnamed in study (the study renders them via helpers/classes, not
`data-component` regions).

**Text is under everything below.** Wherever a Composed-of column names a word, a label, a
count, a path, a timestamp or prose, that string is rendered by the Text atom — the tables
name the copy, not a second way of setting type. The only exceptions are the two primitives
whose `asChild` hands their children to a Slot (Badge, Button); those compose the role class
from Text's map instead of wrapping.
